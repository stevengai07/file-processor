# 智能批量文档信息提取系统 - 产品需求文档

> 版本：MVP 1.0  
> 日期：2026-06-09  
> 文档依据：现有代码能力 + 新增业务需求

## 1. 产品概述

本系统面向需要从大量格式相似或内容相关的文档中收集结构化信息的业务人员。用户先上传一份 Excel 字段模板，系统读取其中定义的关键字段；随后用户可在前端批量上传 PDF、DOCX 等文档，系统逐份提取字段值，允许用户检查和修正结果，最后汇总导出为 Excel 或 DOCX。

现有系统已经具备单个 PDF/DOCX 的文本提取、扫描 PDF OCR、AI 结构化提取和单文件多格式导出能力。本次 MVP 的重点是将固定字段改造成模板驱动字段，将单文件流程改造成前端批量任务，并提供面向多文档的汇总结果。

## 2. 产品目标

1. 用户无需修改代码即可通过 Excel 定义需要提取的字段。
2. 用户可以在一次任务中上传并处理多份文档。
3. 每份文档的结果使用同一字段结构，便于比较、校对和汇总。
4. 部分文档处理失败时，不影响其他文档完成。
5. 用户可以将最终结果导出为一份 Excel 或 DOCX。

## 3. 非目标

- MVP 不包含用户注册、组织、角色和权限管理。
- MVP 不包含长期文档归档、全文检索和版本管理。
- MVP 不自动训练专属模型。
- MVP 不承诺识别手写内容。
- MVP 不处理受密码保护或损坏的文件。

## 4. 目标用户

| 用户 | 典型场景 | 核心诉求 |
|---|---|---|
| 业务操作员 | 从合同、报告、申请材料中整理字段 | 快速批量提取并修正 |
| 数据整理人员 | 将非结构化文档转换成表格 | 字段统一、便于导入其他系统 |
| 审核人员 | 检查自动提取结果 | 能定位来源、识别缺失和低置信结果 |

## 5. 核心流程

1. 用户创建批量提取任务。
2. 用户上传 Excel 字段模板。
3. 系统解析模板并展示字段定义，用户确认或修正。
4. 用户批量上传待处理文档。
5. 用户选择模型、OCR 语言和图像预处理方式。
6. 系统逐份执行文本提取、OCR 和 AI 字段抽取。
7. 前端展示任务进度、成功、失败和待检查数量。
8. 用户在结果表中检查并编辑字段值。
9. 用户导出所有成功结果为 Excel 或 DOCX。

## 8. MVP 功能要求

### 8.1 字段模板

- 接受 `.xlsx` 文件。
- 默认读取第一个工作表。
- 至少支持字段名称、字段说明、数据类型、是否必填四项定义。
- 解析后必须由用户确认，不能直接开始批量处理。
- 字段名称在同一模板内必须唯一。
- 模板至少包含一个有效字段。

### 8.2 文档输入

- 前端支持一次选择或拖入多个文件。
- MVP 支持 `.pdf`、`.docx`。
- 扫描 PDF 自动进入 OCR 流程。
- 单文件失败不得中止整个任务。
- 文件上限、单文件大小上限和任务总大小上限为 `[TBC]`。

### 8.3 字段提取

- 系统只能依据模板字段和文档内容返回结果，不得虚构信息。
- 找不到的字段返回空值，并标记为“未找到”。
- 每份文档必须保留原始文件名和处理状态。
- 必填字段缺失时，文档状态为“待检查”，但仍保留提取结果。
- 日期、数字和布尔值应按模板类型进行格式校验。

### 8.4 人工校对

- 用户可以编辑所有提取字段。
- 人工修改后的值应标记为“已人工修改”。
- 用户可以按成功、待检查、失败筛选文档。
- 失败文档可以单独重试。

### 8.5 导出

- Excel：默认一份文档对应一行，一项模板字段对应一列。
- DOCX：默认一份文档对应一个章节，以字段名和值的形式展示。
- 导出内容必须使用用户确认后的字段顺序。
- 导出应包含原始文件名和处理状态。
- 失败文档默认不进入业务数据区，但在“处理记录”中列出。

## 9. 全局业务规则

| 编号 | 规则 |
|---|---|
| BR-01 | 字段模板确认后才能上传并启动批量提取。 |
| BR-02 | 同一任务内所有文档使用同一份字段模板。 |
| BR-03 | 模板修改后，已产生的结果不会静默覆盖；用户必须确认重新提取。 |
| BR-04 | 系统按文件独立保存状态，部分失败不影响整个任务。 |
| BR-05 | AI 输出必须经过结构校验，字段不得超出模板定义。 |
| BR-06 | 必填字段为空、类型校验失败或抽取异常时，结果进入“待检查”。 |
| BR-07 | 用户修改值后，导出使用修改后的最终值。 |
| BR-08 | 删除任务或原始文件的生命周期策略为 `[TBC]`。 |

## 10. 非功能要求

### 性能

- 提交任务后 2 秒内展示任务已创建和首个进度状态。
- 前端至少每 2 秒刷新一次运行中任务进度，或使用服务端推送。
- 目标批量规模为 `[TBC：建议 MVP 先支持每任务 100 个文件]`。

### 稳定性

- 任务处理必须记录每个文件的错误原因。
- 后端重启后任务和结果是否需要恢复为 `[TBC]`；若需要，必须引入持久化任务存储。
- 导出失败不得删除已提取结果。

### 安全与隐私

- API 密钥不得显示完整值，不得写入日志。
- 临时文件在处理结束后必须删除。
- 日志不得记录完整文档正文。
- 文档保留时间、部署环境和数据合规要求为 `[TBC]`。

### 可用性

- 任务状态、失败原因和缺失字段必须使用中文明确展示。
- 批量处理中用户可以离开结果区域，但是否允许关闭浏览器后继续处理取决于持久化方案。
- 长字段值在表格内应支持展开查看，不能破坏列布局。

## 11. MVP 验收标准

1. 上传一个含 5 个合法字段的 Excel 模板后，页面正确显示 5 个字段及其顺序。
2. 一次上传 10 份 PDF/DOCX 后，系统为每份文件显示独立处理状态。
3. 其中一份文档处理失败时，其余文档仍能完成。
4. 每份成功文档返回与模板完全一致的字段集合。
5. 必填字段缺失的文档自动进入“待检查”。
6. 用户编辑任意字段后，刷新结果视图仍能看到修改值。
7. 导出的 Excel 每份文档一行、字段一列，值与页面最终值一致。
8. 导出的 DOCX 按文档分节，字段顺序与模板一致。
9. 扫描 PDF 可以通过 OCR 参与同一批量流程。
10. 不支持的文件格式在上传阶段被拒绝并说明原因。

## 12. 待确认事项

1. 字段模板的真实列名和示例文件。
2. 是否需要支持 XLS、CSV、图片、TXT 等输入格式。
3. 单次任务文件数量、大小和总容量限制。
4. 是否需要显示字段置信度及原文出处。
5. 是否允许一个文档中存在多条记录，例如一张发票表格含多个明细行。
6. DOCX 导出是汇总报告还是基于用户提供的 Word 模板回填。
7. 是否需要账号、任务历史和长期保存。
8. 文档是否允许发送给第三方云模型，或必须使用本地/私有模型。

## 13. 当前代码差距

| 能力 | 当前状态 | MVP 所需变化 |
|---|---|---|
| 字段定义 | Pydantic 中固定 7 个字段 | 从 Excel 动态生成字段规则和输出结构 |
| 前端上传 | 单文件 PDF/DOCX | 多文件队列、校验、删除和重试 |
| 批处理 | 仅命令行文件夹批处理 | 后端任务接口 + 前端进度 |
| 结果展示 | 单文件固定布局 | 动态字段表格 + 多文档汇总 |
| 导出 | 单文件 JSON/TXT/CSV/DOCX/XLSX | 多文件合并 Excel/DOCX |
| 任务状态 | 无 | 文件级状态、错误、进度和重试 |
| 持久化 | 无 | MVP 可先内存实现；生产环境需要数据库/任务队列 |
| 可运行性 | `agent.py` 存在未解决的 Git 冲突标记 | 开发前必须先修复该文件 |
